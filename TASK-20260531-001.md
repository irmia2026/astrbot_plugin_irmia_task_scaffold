## 需求

`context.register_web_api()` 注册的 Web 路由在 AstrBot (v4.x) 端口 6185 上全部返回
"404 Not found" (HTTP 200)，导致 task_scaffold 的 `/task_scaffold/api/*` 和
`/task_scaffold/dashboard` 无法被浏览器访问。

## 现象

│  `curl http://localhost:6185/task_scaffold/dashboard` → 404 (HTTP 200)
│  `curl http://localhost:6185/task_scaffold/api/current` → 404 (HTTP 200)
│  同样问题复现于 `irmia_heart` 插件的 `/heart/memory/dashboard` 等路由
│  数据层正常：`task_list` / `task_archive` 工具注册后通过 LLM 调用正常工作
│  临时方案：`dashboard-standalone.html`（嵌入数据 + 改写 `get()` 可本地打开）

## 约束

│  M1 · 不要移除 `context.register_web_api` 调用——它应该是正确用法
│  M2 · 修复后需同时验证 irmia_heart 路由是否恢复
│  M3 · 如 AstrBot 版本更换了 API，记录新 API 并迁移两个插件
│  M4 · `dashboard-standalone.html` 是临时产物，修复后可删除

## 排查方向（供参考，非约束）

- AstrBot v4.18+ 是否将 plugin web route 挂载点从 6185 端口移到了其他端口？
- `register_web_api` 的 prefix 参数是否需要在 AstrBot 配置中显式启用？
- 是否需要使用 `@star.web.route()` 等装饰器替代 register_web_api？
- AstrBot 主进程的 Quart/Sanic 路由注册是否在插件加载后才挂载？

## 不改动

- task_list / task_archive / task_scaffold 数据层 — 无需改动
- dashboard.html 模板样式 — 无需改动
- metadata.yaml — 无需改动
